# 전통 카탈로그의 4가지 한계

전통 카탈로그에는 크게 네 가지의 한계가 있다.

아래 네 가지가 바로 그 문제이다.

1. 파티션 리스팅 지옥
2. 파티셔닝 변경 불가
3. 동시 쓰기 데이터 유실
4. 롤백 불가

---

## 1. 파티션 리스팅 지옥

### [문제]

파티션이 늘어나면 늘어날수록 파티션 리스팅 쿼리에 소요되는 시간이 증가하게 되고,
이에 따라 선형적으로 쿼리가 느려진다.

### [왜 생기나]

전통 카탈로그는 해당 데이터가 어떤 폴더에 있는지만 알지,
각 파일 안에 뭐가 들었는지는 알 수 없기 때문이다.

또한 전통 카탈로그의 구조적인 원인으로 인해 N+1 쿼리 문제가 발생하는데,
이해하기 쉽게 아래 예를 들어 설명해보겠다.

**상황: 편의점 점주가 재고를 파악하려고 한다.**

```
1. 점주 -> 직원: "편의점에 어떤 상품들이 있나?"
2. 직원: "삼각김밥, 컵라면, 음료수, 과자, 껌 있어요"
3. 점주: "삼각김밥 재고가 몇 개야?"
4. 직원: 창고 가서 확인 -> "12개요"
5. 점주: "컵라면 재고가 몇 개야?"
6. 직원: 창고 가서 확인 -> "10개요"
   => 이렇게 삼각김밥, 컵라면, 음료수, 과자, 껌 모두 반복
```

총 몇 번 물어봤나?

- 1번: "어떤 상품들이 있어?" (목록 조회)
- 5번: 상품 하나하나 개별로 재고 확인
- 합계: 1 + 5 = **6번**

상품이 100개면? `1 + 100 = 101번`  
상품이 1,000개면? `1 + 1,000 = 1,001번`

이것이 바로 N+1 쿼리 문제다.

### [결과]

이러한 전통 카탈로그의 구조적인 문제를 해결하고자 한 사례가 있긴 했다.
중국의 Zhihu에서 MySQL에서 TiDB라는 분산 DB로 이관하면서
30~40초가 소요되던 파티션 리스팅 쿼리 시간을 6~7초로 개선하였다.

다만, 카탈로그 자체의 구조적인 문제를 원천적으로 해결했다기보다는,
더 성능이 좋은 DB로 갈아끼워서 버텼다고 보는 것이 더 타당하다.

결국, 문제의 본질은 해결되지 않은 것이다.

또한 이 파티션 리스팅 지옥 문제는 클라우드로 오면서 더 악화되었는데,
S3에는 디렉토리 개념이 없기 때문에 전통 카탈로그의 디렉토리 기반 추적이 더 어려워졌다.
S3에 저장된 데이터를 보면 계층 구조가 있는 것처럼 보이지만,
실제로는 LIST API로 흉내만 낸 것이지 사실상 하나의 파일 이름이기 때문이다.

## 2. 파티셔닝 변경 불가

### [문제]

어느 회사에서 처음 파티셔닝을 구상할 땐 `s3://data/events/year=2022` 처럼 연도별로 파티셔닝을 구성해두었다고 하자.
처음엔 잘 사용하다가 데이터 양이 많아지면서 "월별로 파티셔닝을 쪼개야 쿼리가 더 빨라지겠다"는 의견이 나왔다고 해도
전통 카탈로그에서는 파티셔닝을 변경할 수 없어 문제가 발생한다.

### [왜 생기나]

```
현재 상태:
  s3://bucket/events/year=2024/ <- 파일 1,400개

바꾸고 싶은 상태:
  s3://bucket/events/year=2024/month=12/day=01/ <- 파일 50개
  s3://bucket/events/year=2024/month=12/day=02/ <- 파일 50개
  ...
```

폴더 이름 자체가 파티션이기 때문에, 물리적으로 파일을 전부 옮겨야 한다.
그러면 아래와 같은 절차를 거치게 된다.

```
1. 새 폴더 구조로 테이블 새로 만들기
2. 기존 파일 수천 개 전부 읽기
3. 새 구조로 전부 다시 쓰기
4. 기존 테이블 삭제, 새 테이블로 교체
```

### [결과]

작은 데이터면 큰 문제가 되지 않겠지만,
데이터가 수 TB면 수 시간에서 하루까지 소요되고, 그동안 서비스가 중단되는 문제가 발생한다.
서비스를 운영하는 기업에서는 몇십 분만 서비스가 중단되어도 손실이 커지기 때문에, 이는 결코 가벼운 문제가 아니다.

## 3. 동시 쓰기 데이터 유실

### [문제]

아래와 같은 상황을 가정하자.

```
팀원 A가 오전 집계 작업 실행:
  "2024-12-01 데이터 처리해서 저장 중..."

팀원 B가 같은 시간에 수정 작업 실행:
  "2024-12-01 데이터 오류 수정 중..."

결과:
  A의 작업이 B의 작업을 덮어씀 (또는 반대로)
  -> 둘 다 망가진 불완전한 상태
```

### [왜 생기나]

RDBMS(MySQL 등)는 트랜잭션이라는 잠금 기능이 있어서 이를 방지하지만,
전통 카탈로그에는 이러한 안전장치가 없다.
즉, ACID를 지원하지 않기 때문에 발생하는 문제다.

### [결과]

운이 좋으면 데이터가 멀쩡하고, 운이 나쁘면 데이터가 날아간다.
Compaction, 장애 복구처럼 쓰기 작업이 겹치는 순간엔 항상 이 위험에 노출된다.

## 4. 롤백 불가

### [문제]

```
월요일: 데이터 정상
화요일: ETL 버그로 50만 건 오염 저장
수요일: 버그 발견, 월요일 상태로 되돌리고 싶다
        -> 방법 없음. 이미 덮어쓰여짐
```

### [왜 생기나]

전통 카탈로그에는 버전 관리나 스냅샷 기능이 없다.
데이터를 덮어쓰면 이전 상태는 사라진다.

### [결과]

백업을 따로 해두지 않으면 데이터가 완전히 사라진다.
백업이 있더라도 수동으로 복구해야 하기 때문에 그 시간 동안 서비스가 중단되는 문제가 발생한다.

---

## 정리

> 네 가지 한계의 공통 뿌리는 하나다.
> **"전통 카탈로그는 데이터가 어느 폴더에 있는지는 알지만, 각 파일 안에 뭐가 들었는지는 모른다."**
> 이것이 모든 문제의 근원이다.
